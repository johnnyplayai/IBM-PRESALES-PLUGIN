# 00 · 全局约定（单一事实源 / SSOT）

> 本文件是整条流水线的**单一事实源（Single Source of Truth）**。所有产物 schema 常量、术语枚举、命名规则、占位符纪律、防编造红线**只在本文件定义一次**。其余 Agent 提示词（`1`~`4`、`21`、`22`、输出格式、版式定义）一律**引用本表，不得各自写死数字或另立措辞**。
>
> 维护规则：任何 schema / 枚举 / 命名的变更**只改本文件**，下游自动跟随。下游若与本文件冲突，**以本文件为准**。
>
> 适用范围：`PresalesAgentV4/` 全部文件。

## 1. 产物 Schema 常量（页面内容）

| 常量 | 取值 | 适用 |
|---|---|---|
| **L4 上屏判断** | **恰好 1 句 Action Title** | 完整、有观点、可行动；不写字段名"核心结论" |
| **L4 上屏内容块数** | **2-5 块** | 每块只承载一个业务判断/动作/结果 |
| **每块上屏文字** | **1 个短标题 + 1-3 条短句** | 单条短句建议 12-28 字，最长不超过 35 字 |
| **每页上屏总字数** | 依信息密度档位控制 | 见第 2.2 节；不得把讲稿长段搬上屏 |
| **讲稿/论证材料** | 放入 `.讲稿.md` 或 `.内部.md` | 长解释、支撑事实、论证链、说服策略不进入 `Pxx-Pxx.md` |

> **强约束自检（机器可检）**：L4 正文页必须有 1 句 Action Title；上屏内容块数 ∈ [2,5]；每块短句数 ∈ [1,3]；单条上屏短句 ≤35 字；不得出现 `主张句` / `支撑点` / `论据块` / `核心结论` 等字段名作为上屏结构。不符即结构不达标，须修正。
>
> **下游一致性**：`3_方案输出.md` 负责按密度生产可上屏短稿；`4_页面输出.md` 只消费短稿并做视觉转译；`输出格式-Lovart-GPTImage2.md` 与 `页面版式定义-IBM.md` 不得要求下游重新扩写正文。

## 2. 术语与枚举（跨 Agent 状态）

### 2.1 咨询基调（唯一权威枚举）

全流水线只用以下两个值，**废弃"风格 A/B""风格"等旧措辞**：

- `管理咨询` —— McKinsey / BCG / Bain 取向：业务价值与战略先行，技术术语克制，强 SCQA + 议题树，价值池/ROI/NPV 为高潮，视觉偏 2×2 矩阵/价值树/瀑布。
- `数字化转型` —— Accenture / Deloitte / SAP 生态取向：现状能力→目标能力的转型路径，业务与 IT 双线，可出现 S/4HANA、BTP、Clean Core 等但须翻译业务影响，视觉偏能力地图/架构分层/路线图/核心+外围放射。

**页面标记格式（唯一写法）**：每页顶部携带 `> 咨询基调：管理咨询`（或 `数字化转型`）。**只能取这两个值之一，不写 A/B、不加"风格"后缀、不与信息密度档位混写。**

### 2.2 信息密度档位（由 Solution 阶段决定）

信息密度是内容产品的颗粒度，**由 `3_方案输出.md` 在生成 `02_方案输出/` 前询问确认并固化到 `Agenda.md` 前置共识**；`4_页面输出.md` 不再询问、不再改写密度，只读取上游已定稿的上屏短稿并转成 Lovart 提示词。密度仅控制 L4 正文页；L1 封面 / L2 目录 / L3 章节页恒定完整。

1. `高密度（专家评审）`：适合技术/方案评审。每页 4-5 个内容块，每块 2-3 条短句；可保留更多产品名、数字、边界条件，但仍必须是上屏短句，不写长段论证。
2. `中密度（正式汇报，默认推荐）`：适合客户正式汇报。每页 3-4 个内容块，每块 1-2 条短句；只保留直接支撑本页判断的事实、数字和关键词。
3. `低密度（高管沟通）`：适合高管沟通。每页 2-3 个内容块，每块 1 条短句；一页只讲一个判断，强调留白和视觉锚点。

**优先级唯一声明**：密度档位在 Solution 阶段生效，覆盖任何下游"完整继承 / 原文输出 / 不简化"表述。页面输出不得把 `.讲稿.md`、`.内部.md` 或长解释重新搬进画面。

### 2.3 不可丢失硬核清单（任何档位下必须处理）

数字 / 来源标注 / SAP 模块名与产品名 / 客户专有名词与全称 / 价值落点（杠杆·指标·量级）/ 本页 Action Title 的核心判断。**硬核清单不等于全部上屏**：能进上屏短句的保留在 `Pxx-Pxx.md`；需要解释的放入 `.讲稿.md`；内部策略放入 `.内部.md`。低密度下也不得删除事实依据，只是改变归属。

### 2.4 主体称呼与正式名称（全流水线强约束）

输出给人阅读的产物（资料整理 md、对客 PPT 上屏、Lovart 提示词、作战手册、技术标 Word 正文）必须使用正式主体名称，不使用泛化代称：

- **供应商默认名称**：本 Presales Agent 默认供应商为 `IBM`。如输入资料或用户明确给出投标主体全称/简称（如 `IBM Consulting`、`国际商业机器（中国）有限公司`），优先使用该正式名称；缺失时用 `IBM`，不得写"我方"。
- **客户名称**：从输入资料、RFP、会议纪要或用户说明中提取客户正式名称。已知全称时使用全称或"全称（简称）"；正文后续可用简称，但不得写"客户方""贵方""对方"等泛称。**客户正式名称缺失是硬停止点**：必须停止当前阶段，不生成、不写入任何下游产物，并要求用户提供客户正式名称；严禁写 `【待确认：客户正式名称】` 或用"客户"硬代。
- **内部/外部一致**：内部作战手册也要写"IBM 投标团队""IBM 商务数据""IBM 责任边界"等明确主体；对客产物更要避免第一人称与甲乙方口语。
- **路径例外**：既有目录名、文件名、引用路径（如 `04_我方素材`）保持不变，因为它们是流水线结构约定；但正文解释这些路径时，应转写为"IBM 素材/供应商素材"等正式称呼。

## 3. 双产物约定（对客上屏 / 内部情报的物理隔离）

为把"内外分离"从"靠模型自觉"升级为"物理不可达"，方案输出的正文页**拆成两份产物**：

- `02_方案输出/Pxx-Pxx.md` —— **仅上屏短稿**：面包屑、Action Title、2-5 个短内容块、可上屏关键数字/标签、视觉逻辑。不得写长段论证，不得出现 `主张句` / `支撑点` / `论据块` / `核心结论` 字段名。
- `02_方案输出/Pxx-Pxx.讲稿.md` —— **可选讲稿/论证材料**：长解释、支撑事实、来源、讲者备注、待澄清问题。下游页面输出默认不读取。
- `02_方案输出/Pxx-Pxx.内部.md` —— **仅内部情报**：战略意图、签约卡点、门槛人、决策链、说服/安抚策略、竞争态势等。**永不上屏。**

> **下游读取铁律**：`4_页面输出.md`（页面出图）与任何对客产物 Agent **只读 `Pxx-Pxx.md` 与 `骨架.md`，严禁读取 `.讲稿.md` 与 `.内部.md`**。骨架页（封面/目录/章节）写入 `骨架.md`，无内部分身。
>
> 内部情报的**唯一合法归属**是 `.内部.md`。判据：若某句话当着客户念出会让客户尴尬、或感到被"分析/算计/搞定"，它只能进 `.内部.md`，绝不进 `Pxx-Pxx.md`。

## 4. 防编造红线（全流水线统一）

### 4.1 三类编造禁区（无本地源、联网也无公开源 → 用占位符，严禁编造或推测）

| 类别 | 具体内容 |
|---|---|
| 客户决策与预算 | 决策人投票权重、预算金额、审批流程、授权额度 |
| 竞品内部信息 | 竞品报价、投入人天、客户承诺、团队履历 |
| 事实性纪要 | 会议已发生发言、邮件原文、待办承诺人与截止日 |
| IBM/供应商商务数据 | 报价、人天、单价、注册资本、顾问人数、bench |

### 4.2 占位符纪律（统一约定）

- `【待补：xxx】` = 需 **IBM/供应商项目团队**填入真实数据。
- `【待确认：xxx】` = 需向**客户正式名称所指组织/内部相关方**澄清。

### 4.3 数字与价值表达（**不写免责声明**）

- 来自输入资料 / 联网可靠来源：直接引用并标来源（脚注 `[^N]` + 完整 URL + 访问日期）。
- **不确定的量化只给数量级**（如"周转天数下降约一个量级到十几天级别""节省百万级成本"），**不写带具体百分比+免责声明的伪精确数字**（禁止 `-10%~-20%（行业经验假设）` 这类"带免责声明的编造"）。
- 凡能锚定来源/类比对象的，给区间并标来源；**完全无依据的，只给定性方向或数量级，不编造精确数字、不靠"行业经验假设"免责兜底。**

### 4.4 SAP 产品/模块名单一事实源

凡引用 SAP 产品/模块名，**必须出自** `01_资料整理/03_需求范围/产品模块对齐表.md`。引用表外模块名一律标 `【待确认：模块名不在对齐表】`，**禁止静默通过（不采用 fail-silent）**。联网核验限供应商官方域（`sap.com` / `help.sap.com`）。

## 5. 维度词 / 枚举词禁令（一处定义，下游引用）

**禁止**把以下词作为上屏标题、块标题、分栏标题、标签或任何画面文字：

- 分析维度词：`现状` / `挑战` / `数字化对策` / `差异化` / `优势`。
- 结构枚举词：`模块` / `模块 A` / `模块 1` / `模块 2`。
- 字段名：`核心结论` / `论据块` / `价值落点` / `战略意图`。

**正向脚手架（业务化小标题命名公式）**：围绕「业务对象 + 能力动作 / 价值结果 / 决策结论」命名，写成一个具体业务判断。

| 反例（禁用） | 正例（业务化命名） |
|---|---|
| 现状 | 订单交付周期被手工对账拖到 5 天 |
| 数字化对策 | 实时仓储调度把缺货响应压到分钟级 |
| 差异化/优势 | 全球模板治理让 30 国账套 1 套规则跑通 |
| 模块 A | 财务共享中心的自动对账与合并 |
| 模块 1 | 集团成本核算的多维实时归集 |

> 块的编号只存在于思考过程，不进入输出文本与画面。自检时**不是"找禁词"，而是逐条问"这个小标题是否是一个具体业务判断/动作/结果"**。

## 7. 自检设计原则（可验证优先）

自检项必须是**模型能真正执行的存在性/格式检查**，不写模型无法验证的定性判断：

- ✅ 好的自检："每个量化数字旁是否都带 `[来源]` 或数量级表述？（无标记即不合格）"
- ❌ 差的自检："关键论点是否有 Benchmark 支撑？"（模型无法在不重新联网下判断真伪，必自评通过）

凡涉及"是否真实/是否充分/是否预判"的定性项，一律改写为"是否带某标记/是否含某字段/是否逐条对应"的存在性检查。

## 8. 停止点契约（机器可检）

凡提示词要求"停下来等用户确认"的环节，**用输出契约而非否定式口号**实现。模板：

> **本回合输出契约（机器可检）**：本回合回复**只允许**包含 [允许的内容]，**不得出现** [禁止的标志物，如某标题/某表格/某文件写入动作]。若发现自己正要产出禁止物，立即停止并改为请用户确认。

各 Agent 在停止点引用本契约模板，填入各自的"允许/禁止"清单。

## 10. 文风自然化（去 AI 味 · 全员适用）

凡产出会被人读到的文字——上屏 PPT 文字、技术标 Word 正文、作战手册、资料整理内容 md——都要避免"AI 腔"，读起来像资深顾问 / 技术主笔手写，不像模型生成。



### 10.1 五条核心原则（速查）

1. **删填充与开场白**：去掉"为了实现这一目标""值得一提的是"这类铺垫拐杖，直接说事。
2. **破公式结构**：不做二元对比设问、不做戏剧性分段、不强凑排比。
3. **变节奏**：长短句结合，连续三句同构就打断一句；段尾收法多样，别每段都用同款短句收口。
4. **信任读者**：事实直陈，不软化、不反复铺垫、不手把手解释显而易见的东西。
5. **删金句**：读起来像可裱起来的口号 / 标语，就重写成平实陈述。

### 10.2 内容模式（夸大与空话）

- **夸大象征意义 / 宣传腔**：删"标志着 / 见证了 / 是……的体现 / 彰显其重要性 / 谱写新篇章 / 赋能 / 保驾护航 / 一站式 / 全方位 / 打造闭环"。不替任意事实硬安"更宏大的意义"。
  - 病句："本方案标志着客户数字化转型的关键里程碑。" → "本方案覆盖采购、生产、财务三个流程域，替换现有 N 套系统。"
- **知名度 / 资历堆砌**：不堆"行业领先 / 众多头部客户 / 广泛认可"等无上下文的资历罗列；要写就给一个具体、可核的事实。
- **拖尾式肤浅分析**：警惕句尾用"……，从而实现了……""……，确保了……""……，进一步提升了……""……，体现了……"硬凑深度。直接陈述事实，砍掉这条尾巴。
- **模糊归因**：不写"业内普遍认为 / 实践证明 / 相关研究表明 / 专家指出"这类无源断言。要么给具体来源（对齐表、RFP §条款、IBM 素材、联网信源带 `[^N]`），要么按第 4 节标 `【待补：来源】`。既是 AI 腔，也是编造风险。
- **"挑战与展望"套路**：不套"尽管面临诸多挑战……但凭借……必将……"的公式段。讲约束就讲具体约束与应对，不用范式化收尾。

### 10.3 语言与语法模式

- **AI 高频词 / 过渡词**：`值得一提的是` / `综上所述` / `总而言之` / `不难看出` / `可以说` / `众所周知` / `此外` / `随着……的不断深入` / `在……的今天`。能用"还 / 并 / 因此 / 同时"就不用"此外"，能删的过渡句直接删。
- **系动词回避**：别用"作为 / 代表 / 充当 + 一个……"绕开"是"。能写"X 是 Y"就别写"X 作为一个 Y 而存在"。
- **否定式排比**：`不仅……而且……` / `这不仅是……更是……` / `不是……而是……`，过度使用即模板腔，还原成平直陈述。
- **三段式 / 排比强凑**：不为"显得全面"把每个观点凑成三点、每句用"既…又…还"。该一点就一点，该两点就两点。
- **同义词循环**：同一主体短距离内不必换着花样指代（"该方案 / 本系统 / 此平台 / 这套架构"轮番上场）；指代清楚比花哨重要。
- **虚假范围**：不用"从底层架构到顶层应用 / 从数据到决策"这种伪全景排比，除非两端确在同一有意义的尺度上。

### 10.4 风格模式（标点与排版）

- **破折号克制（重点）**：不用破折号（`——` / `—`）串接句子或堆叠补充。改用句号断句、冒号引出、把次要补充放进括号。**每页上屏 / 每段 Word 正文，破折号至多 1 处，能不用就不用。**（范围号 `~`、连字符 `-`、`数字-数字` 不在此列。）
- **加粗克制**：加粗只给真正的关键词或关键数字，不整句整段加粗；正文里避免"**用户体验：** ……""**性能：** ……"这种粗体标题 + 冒号的清单堆叠。
- **去对称癖**：短内容块之间不必字数对齐、句式对仗。
- **空泛副词与强调词**：删 `极大地` / `显著地` / `全面` / `有效` / `进一步` / `不断` / `更好地` / `高度` / `充分`，用具体事实、动作或数量级替代。
- **不用表情符号**装饰标题或要点（对客件尤忌）。

### 10.5 填充与回避

- **填充短语**：`为了实现这一目标`→`为此`；`在这个时间点`→`现阶段`；`具有……的能力`→`能 / 可`；`由于……的事实`→`因为`。
- **过度限定**：`可能潜在地或许会在一定程度上……`一串模糊限定叠加，压成一句确定的话；真不确定就按第 4 节给数量级或标占位符，不写免责式伪精确。
- **空泛积极收尾**：删 `未来可期` / `前景广阔` / `迈向新台阶` / `谱写新篇章` / `驶入快车道`。收尾给具体结论、下一步或可验证的价值落点。

### 10.6 不采用的 Humanizer 做法（边界）

Humanizer-zh 面向博客 / 评论等个人化写作，其下列建议**与本项目正式件定位冲突，不要做**：注入个性、表达主观好恶、多用第一人称"我 / 我觉得"、承认复杂或矛盾的感受、刻意跑题与半成型想法、写口语化金句。本项目去 AI 味只做"减法"（去机器腔），不做这类"加个性"的"加法"。技术标、方案、作战手册一律保持客观专业。

### 10.7 自检（存在性检查）

输出前逐段问："这段像人写的还是像模型生成的？"重点过六样：① 破折号是否超约定上限（10.4）；② 是否出现 10.3 套话 / 过渡词；③ 是否强凑三段式或否定式排比（10.3）；④ 是否有夸大象征 / 宣传腔 / 拖尾式分析（10.2）；⑤ 是否有模糊归因（10.2）；⑥ 是否有填充短语或空泛积极收尾（10.5）。命中即改写。
