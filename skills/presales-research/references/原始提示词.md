# 资料整理

> 本文件是**通用提示词**。将 `00_前期输入/` 里的原始素材（邮件、会议纪要、客户文档、RFP 等）整理成结构化的售前素材库，落位到 `01_资料整理/` 下，供下游 `3_方案输出.md` 和 `02_方案输出/` 调用。
>
> 产品/行业/公司无关——任何 B2B 售前或咨询项目只要遵循同样的目录骨架，这份指导都适用。具体产品线、行业术语、本地化条目由各项目自行注入到 `00_前期输入/` 的素材里。
>
> **本文件遵循 `00_全局约定.md`（SSOT）**：占位符纪律、三类编造禁区、命名/路径规则、SAP 模块名单一事实源、主体称呼与正式名称等全局约束**只在 SSOT 定义一次**，本文件只留一句摘要 + 指针，不重复完整定义。如本文件与 SSOT 冲突，**以 SSOT 为准**。客户正式名称缺失是硬停止点：必须停止资料整理并要求用户提供正式名称，不生成、不写入 `01_资料整理/`。

## 1. 任务概述

### 1.1 输入
- **目录**：`00_前期输入/`
- **内容**：原始素材文档（.md / .pdf / .docx / .xlsx / .txt ...），无固定格式
- **清单**：由 `00_前期输入/Processed_Files.md` 维护"已处理文件索引"

### 1.2 输出
- **目录**：`01_资料整理/`
- **结构**：6 个主题子目录，每个子目录含若干内容 md（**不生成 `index.md`**）
- **文件清单的唯一事实源**：第 2 节的目录结构清单。**实际产出文件必须与第 2 节逐一对应（不多不少）**；不在本文件硬编码"共 N 个文件"的魔数，避免清单与数字脱节。
- **组织原则**：按内容主题分类（不按交付场景分目录），场景适配通过各内容 md 的 frontmatter（适用场景 / 重要度）、摘要段和顶层"场景入口导航"完成

### 1.3 适用场景
生成的素材同时支撑以下客户交流与下游生成动作：
- **首次交流**——公司介绍 / 能力展示 / 建立认知
- **正式投标**——RFP 应答 / 技术方案 / 商务报价
- **后期补充**——客户追问应答 / 资料补交 / 方案迭代
- **SAP 方案输出**——供 `3_方案输出.md` 生成 SCQA、Agenda、正文页和 Lovart 页面转译输入

## 3. 场景入口导航

下游组装方案时，按场景从下列入口开始读，再按各文件 frontmatter（适用场景 / 重要度）与摘要段判断是否深入：

| 场景 | 起始入口 | 重点目录 |
|---|---|---|
| **首次交流** | `04_我方素材/公司介绍要点.md` + `01_客户画像/客户档案.md` | 04 + 01 + `02_项目情报/项目概况.md` |
| **正式投标** | 按 03 → 04 → 01 → 02 → 05 → 06 顺序全量通读 | 03 + 04 + 01（全量通读） |
| **后期补充** | `05_沟通记录/问答库.md` | 按客户追问的具体问题反查相关目录 |
| **SAP 方案输出** | `03_需求范围/产品模块对齐表.md` + `01_客户画像/业务与IT现状.md` + `02_项目情报/项目概况.md` | 03 + 01 + 02 + 04 + 06 |

### 3.1 面向 3_方案输出.md 的读取适配

`3_方案输出.md` 的第一阶段需要生成 `02_方案输出/Agenda.md`，第二阶段需要生成高密度页面内容。资料整理必须提前沉淀下游所需的叙事、事实、模块、价值和视觉线索。

> **frontmatter 被下游消费**：下游 `3_方案输出.md` 在筛选输入时，**优先读取 frontmatter「适用场景」含「SAP 方案输出」、且「重要度」为 ★★★ 的文件**。因此凡是支撑 SAP 方案输出的核心 md，必须在 frontmatter 正确标注「适用场景」与「重要度」，否则会被下游漏读。

| 下游任务 | 资料整理必须提供的输入 | 推荐来源文件 |
|---|---|---|
| 汇报对象与叙事判断 | 听众画像、决策关注点、客户目标、项目背景 | `01_客户画像/关键人物.md`、`01_客户画像/客户档案.md`、`02_项目情报/项目概况.md` |
| SCQA 故事线 | Situation / Complication / Question / Answer 可用素材 | `01_客户画像/业务与IT现状.md`、`02_项目情报/项目概况.md`、`03_需求范围/业务需求要点.md`、`06_外部参考/行业背景.md` |
| PPT 目录设计 | 可拆成章节的主题、每个主题的业务目的、推荐表达方式 | 各内容 md 的摘要段 + frontmatter（适用场景 / 重要度）+ `03_需求范围/产品模块对齐表.md` |
| 正文页核心论点 | 现状/挑战、数字化对策、差异化优势、业务价值 | `03_需求范围/*`、`04_我方素材/*`、`06_外部参考/技术补充.md` |
| SAP 模块与架构 | 产品白名单、部署形态、商业框架、License 计量、扩展性模型、集成边界、架构原则、最新能力核验 | `03_需求范围/产品模块对齐表.md`、`03_需求范围/技术范围.md`、`06_外部参考/技术补充.md` |
| 价值页与 ROI/TCO 讨论 | 已有数字、可量化价值、行业经验假设、不可直接使用的数据 | `02_项目情报/项目概况.md`、`04_我方素材/相关案例库.md`、`06_外部参考/行业背景.md` |
| Lovart 页面转译 | 推荐图表类型、页面逻辑关系、可视化对象、页面重点 | 各内容 md 的"下游方案输出提示"段 |

### 3.2 下游消费原则

- 下游 `3_方案输出.md` 会读取 `01_资料整理/` 下所有资料，因此每份内容 md 的 frontmatter（适用场景 / 重要度）与摘要段必须支撑快速筛选，避免下游全量阅读后仍无法判断重点。
- `03_需求范围/产品模块对齐表.md` 是 SAP 产品、模块和能力名称的**单一事实源（详见 SSOT `00_全局约定.md` §4.4）**；下游引用 SAP 模块时必须优先来自本表。
- 每份内容 md 必须在"下游方案输出提示"中给出可用于 SCQA、Action Title、页面论点和视觉表达的素材。
- 对下游不能直接使用的信息，必须按 SSOT 占位符纪律标注（详见第 6 节摘要），不得编造。
- 对页面设计有帮助的结构化关系（因果、递进、对比、闭环、分层、能力地图、价值树、泳道流程等）应提前写入内容 md，供第二阶段页面详细设计调用。

### 4.1 强制字段
- **frontmatter**：5 个字段全部必填
  - **适用场景**：从 `首次交流 / 正式投标 / 后期补充 / SAP 方案输出` 中选 1-4 个
  - **重要度**：`★★★` 核心必备 / `★★` 重要参考 / `★` 补充性
  - **注意**：「适用场景」与「重要度」会被 `3_方案输出.md` 用于筛选（见 §3.1），标注务必准确
- **摘要段**（开头 `>` 引用块）：必须有，3-5 行
- **下游方案输出提示**：必须有，说明该文件如何支撑 `3_方案输出.md` 的 SCQA、Agenda、正文页和视觉逻辑
- **信源段**：必须有；分"本地 / 联网"两类；联网条目用脚注 `[^N]` 在正文引用
- **更新日志**：每次修订追加一行，不删旧记录

### 4.2 人工编辑保护
- 用 `<!-- MANUAL:START --> ... <!-- MANUAL:END -->` 包裹
- AI 遇到此标记**整段跳过**，不读、不改、不删
- 允许在文件任意位置插入多个 MANUAL 块
- 若一个原 AI 生成的段落被人工编辑但未加标记，AI 在下次增量更新时**应当保守**：检测到内容偏离当前输入源时，追加新版本而非覆盖旧版本，并在更新日志注明

## 5. 各内容 md 的职责表

逐文件说明"收什么内容、推荐章节骨架、重点关注"。文件清单以第 2 节为准。各项目在实施时按自身产品栈、行业、区域补足具体术语与条目。

### 01_客户画像

#### `客户档案.md`
- **内容**：公司背景、发展历程、业务线/产品、全球布局、主要生产/运营基地、股权/集团关系、财务体量
- **章节骨架**：公司概况 / 业务版图 / 地域布局 / 产品与客户 / 组织与股权 / 财务体量
- **下游重点**：支撑 SCQA 的 Situation、封面/开场页的客户背景、高管汇报中的业务上下文
- **联网**：官网、公开财报、新闻；涉及股权和财务数据如本地未提、网上也找不到，按 SSOT 占位符标注

#### `关键人物.md`
- **内容**：决策链（Sponsor / Decision Maker / Champion / User / Coach）、姓名/职位/背景、介绍人关系、沟通历史中已出现的联系人
- **章节骨架**：决策链图 / 关键人物画像（逐人）/ 介绍人与关系网 / 沟通记录索引
- **下游重点**：支撑 `Agenda.md` 中的汇报对象与叙事判断、页面术语深度、价值表达侧重
- **禁区**：决策权重、预算授权额度若本地未提，不得推测（SSOT §4.1「客户决策与预算」）

#### `业务与IT现状.md`
- **内容**：现有系统栈（ERP / CRM / MES / 其他）、业务流程痛点、信息化成熟度、自动化与设备现状、已识别的改进诉求
- **章节骨架**：现有系统地图 / 业务流程痛点（按模块）/ 自动化与集成现状 / 信息化成熟度评估 / 改进诉求清单
- **下游重点**：支撑 SCQA 的 Complication、正文页"现状/挑战"、架构页的 As-Is 对比

### 02_项目情报

#### `项目概况.md`
- **内容**：项目名称、范围（模块/地域/人数）、分期上线计划、预算线索、关键里程碑、官方启动时间
- **章节骨架**：项目定位 / 范围边界 / 里程碑 / 预算线索 / 组织分工
- **下游重点**：支撑 PPT 章节划分、项目范围页、路线图页、实施阶段页

#### `采购与节奏.md`
- **内容**：选型流程（候选→筛选→决策）、各阶段时间点、RFP 发放计划、决策机制（招标 / 单一采购 / 谈判采购）、IBM 在流程中的位置
- **章节骨架**：选型流程图 / 关键时间点 / 决策机制 / IBM 位置与行动节奏
- **下游重点**：支撑汇报策略、投标节奏、方案重点取舍和客户下一步行动建议

#### `竞争情报.md`
- **内容**：已知竞争对手清单（及其已知优劣势）、客户对竞品的评价、IBM 差异化点、同类项目中竞品的打法
- **章节骨架**：竞品清单 / 竞品画像（逐家） / 客户评价 / IBM 差异化对标 / 应对策略
- **下游重点**：支撑差异化优势页、竞争对标页、IBM 能力强调顺序
- **禁区**：不得凭空推测竞品报价或人员配置；联网只取公开信息（SSOT §4.1「竞品内部信息」）

### 03_需求范围

#### `业务需求要点.md`
- **内容**：按业务域/模块（按所售产品线的标准模块划分）梳理业务诉求，每条标注来源和优先级
- **章节骨架**：按模块分章 / 每条需求含"原文摘录 → 解读 → 优先级 → 来源"
- **下游重点**：支撑正文页"现状/挑战"和"数字化对策"，为每页核心论点提供事实依据

#### `技术范围.md`
- **内容**：产品形态与部署模式（on-prem / Private Edition / Public Edition / SaaS）、核心模块组合、集成边界（与客户既有系统、外部机构、自动化设备、第三方服务）、数据迁移范围
- **章节骨架**：核心产品栈 / 部署形态 / 集成清单 / 数据迁移 / 技术假设
- **与对齐表的关系**：本文件的部署形态、模块组合**必须与 `产品模块对齐表.md` 保持一致**；以对齐表为准
- **下游重点**：支撑 SAP 架构页、集成页、部署模式页和技术路线页

#### `合规与本地化.md`
- **内容**：从本产品栈的"本地化/合规特性清单"中**筛选**与本项目强相关的条目，重点标注 ⚑ 项；按合规主题归类（税务 / 海关 / 外汇 / 数据合规 / 行业监管 / 其他）
- **章节骨架**：按合规主题分章 / 每项含"功能说明 / 本项目相关性 / 实施要点 / 参考来源"
- **SAP 本地化抓手（强制）**：
  - **对照 SAP 官方本地化覆盖矩阵逐项核对**——把"**目标国家在目标部署形态（尤其 Public Edition）下是否被官方支持**"作为**强制判断项**。某项本地化在 on-prem/Private 支持、但 Public Edition 未覆盖，是真实存在的差距，必须显式标出，不得默认"都支持"。
  - 核对维度示例：法定财务报表 / 税务申报 / 电子发票 / 外汇 / 行业监管报送 是否随部署形态而有覆盖差异。
  - **中国项目预置核对清单**（如项目落在中国，至少逐项核对）：
    1. **全电发票（数电票）对接** —— 与金税系统/数电票平台的对接方案与覆盖状态；
    2. **国标会计科目表与财务报表** —— 中国会计准则科目表、法定财务报表（资产负债表/利润表/现金流量表）输出；
    3. **金税 / 数电票** —— 增值税发票开具与认证、进销项对接；
    4. **数据出境（PIPL / 数据跨境）对 RISE 公有云部署的影响** —— Public Edition 的**境内数据中心要求**、个人信息出境合规对部署区域的硬约束；
    5. **本地化 add-on** —— 税务对接通常依赖第三方 add-on（如东软、百望税务对接），需明确"标准产品覆盖到哪、哪部分靠 add-on 补"。
  - 凡涉及国家级官方覆盖状态的判断，需标来源（SAP 官方本地化矩阵 / `help.sap.com`），无来源则按 SSOT 占位符标 `【待确认：xxx】`。
- **下游重点**：支撑合规价值页、本地化能力页、风险控制页

## 7. 增量更新流程

### 7.1 `00_前期输入/Processed_Files.md` 规格

每次整理前读、处理后写的"账本"。格式：

```markdown
# 已处理文档索引

> 由 `2_资料整理.md` 自动维护。每次整理前先读本文件，处理后更新。

## 文件清单

| 文件 | 首次处理 | 最近处理 | 内容指纹 | 关联输出 |
|---|---|---|---|---|
| 00_前期输入/产品本地化清单.md | YYYY-MM-DD | YYYY-MM-DD | sha256:abc1234... | 03_需求范围/合规与本地化.md;<br>03_需求范围/技术范围.md;<br>06_外部参考/技术补充.md |
| 00_前期输入/邮件历史.md | YYYY-MM-DD | YYYY-MM-DD | sha256:def5678... | 02_项目情报/*;<br>05_沟通记录/会议纪要汇总.md |

## 处理历史

- **YYYY-MM-DD** · 初次处理 N 份文档 → 按第 2 节结构清单生成全部内容 md

## 待处理 / 跳过

- （列出已知存在但尚未处理、或主动跳过的文件及原因）
```

> **指纹说明**：`内容指纹` 为输入文件内容的 **sha256**，用于检测文件是否变化。**指纹可由 Claude 直接计算，或用 Bash `shasum -a 256 <file>` 计算**，两种方式取值一致即可。

### 7.2 执行流程

```
1. 读 00_前期输入/Processed_Files.md
   - 若不存在，视为首次执行（空基线）

2. 扫描 00_前期输入/ 当前所有文件
   - 排除 Processed_Files.md 自身
   - 对每个文件计算 sha256 内容指纹（Claude 计算或 Bash shasum，二选一）

3. 对照账本识别三类差异
   a. 新文件（账本中无记录）
      → 分析文件内容判断其主题归属
      → 更新相关内容 md
      → 在账本中新增一行

   b. 指纹变化文件（sha256 不一致）
      → 按账本"关联输出"反查所有受影响的内容 md
      → 对每个受影响 md 做增量更新
      → 更新账本中该行的"最近处理"和"内容指纹"

   c. 已消失文件（账本有记录、当前目录无文件）
      → 在账本"文件清单"表中将该行"最近处理"改为 "已移除 YYYY-MM-DD"
      → 不删除已有输出 md（已产出的内容仍可能有价值）
      → 在处理历史记录该变更

4. 写入所有内容 md
   - 遇到 <!-- MANUAL:START/END --> 区块整段跳过
   - 检测到疑似人工编辑但未加标记的段落，保守追加不覆盖，日志注明
   - 每个被改动的 md 在 ## 更新日志 追加一行
   - 同步更新 frontmatter 的"最近更新"字段

5. 更新 Processed_Files.md
   - 所有变更文件的行更新指纹与时间
   - "处理历史"追加一行本次操作摘要
```

### 7.3 首次执行
- `01_资料整理/` 为空 + `Processed_Files.md` 不存在 → 所有输入文件视为"新文件"，走 7.2 步骤 3.a
- 一次性按第 2 节结构清单生成全部内容 md + 建立 `Processed_Files.md`
